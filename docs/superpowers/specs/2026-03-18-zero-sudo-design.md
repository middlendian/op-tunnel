# Zero-Sudo op-tunnel: Socket Cleanup via `op-tunnel-keepalive`

**Date:** 2026-03-18
**Status:** Draft
**Branch:** TBD (from `further-streamline`)
**Supersedes:** Sections 1, 2, 5, 6, 7, 8 of `2026-03-15-streamline-setup-design.md`

## Problem Statement

The current design (from the `further-streamline` branch) requires `sudo` on every machine where op-tunnel is installed:

1. **Socket directories:** `/opt/op-tunnel/<user>/` requires root to create the base directory.
2. **sshd drop-in:** `/etc/ssh/sshd_config.d/op-tunnel.conf` provides `AcceptEnv LC_OP_TUNNEL_ID` and `StreamLocalBindUnlink yes`. Requires root to install and sshd reload.

This is friction — especially on remote servers the user doesn't administer. The `AcceptEnv LANG LC_*` directive is already present in Ubuntu/Debian's default `sshd_config`, covering `LC_OP_TUNNEL_ID`. The remaining need for `StreamLocalBindUnlink yes` (stale socket cleanup on reconnect) can be handled by a userspace daemon instead of an sshd configuration change.

## Design

### 1. Socket Paths: `/tmp/op-tunnel-$USER/`

Replace `/opt/op-tunnel/<user>/` with `/tmp/op-tunnel-<user>/`:

```
/tmp/op-tunnel-<user>/           # created by user, 0700
/tmp/op-tunnel-<user>/server/    # op-tunnel-server socket (local machine)
/tmp/op-tunnel-<user>/client/    # remote-forwarded sockets (remote machine)
```

No root-owned base directory. Users create their own directories in `/tmp/` without sudo.

**Server socket (local machine):**
`/tmp/op-tunnel-<user>/server/op.sock`

**Client socket (remote machine):**
`/tmp/op-tunnel-<user>/client/<LC_OP_TUNNEL_ID>.sock`

**`/tmp/` volatility:** On reboot, `/tmp/` is cleared. This is acceptable:
- The local server (brew service) recreates its socket directory on startup.
- On remotes, `op-tunnel-keepalive` (see below) creates the client directory on first SSH connection.

**Code changes in `oppath/oppath.go`:**

Replace the `BaseDir` constant with a function:

```go
// UserDir returns the per-user op-tunnel directory.
func UserDir(user string) string {
    return "/tmp/op-tunnel-" + user
}

func ClientSocketPath(user, tunnelID string) string {
    return filepath.Join(UserDir(user), "client", tunnelID+".sock")
}

func ServerSocketPath(user string) string {
    return filepath.Join(UserDir(user), "server", "op.sock")
}
```

### 2. SSH Configuration Changes

**`packaging/ssh.config.tmpl`** — updated paths, remove `StreamLocalBindUnlink`:

```
# Auto-generated by op-tunnel-setup. Do not edit manually.
# Delete ~/.config/op-tunnel/ and re-run op-tunnel-setup to regenerate.
RemoteForward /tmp/op-tunnel-%r/client/@@TUNNEL_ID@@.sock /tmp/op-tunnel-@@LOCAL_USER@@/server/op.sock
SetEnv LC_OP_TUNNEL_ID=@@TUNNEL_ID@@
ServerAliveInterval 30
```

`StreamLocalBindUnlink yes` is removed — stale socket cleanup is now handled by `op-tunnel-keepalive`. `ServerAliveInterval 30` is retained (SSH-level keepalive, independent of op-tunnel).

**`packaging/op-tunnel-sshd.conf`** — removed entirely. Not needed because:
- `AcceptEnv LANG LC_*` is already in Ubuntu/Debian's default `sshd_config`.
- `StreamLocalBindUnlink yes` is replaced by `op-tunnel-keepalive`.

### 3. New Binary: `op-tunnel-keepalive`

A Go binary started by `~/.ssh/rc` on the remote host when an SSH session begins. It handles socket lifecycle: directory creation, stale socket cleanup, and session monitoring.

**Invocation (one line appended to `~/.ssh/rc` by installer):**

```bash
[ -n "$LC_OP_TUNNEL_ID" ] && op-tunnel-keepalive --session-pid "$PPID" --tunnel-id "$LC_OP_TUNNEL_ID"
```

`$PPID` in `~/.ssh/rc` is the sshd session child process — it lives for the entire SSH session.

**Flags:**
- `--session-pid <pid>` — PID of the sshd session process to monitor (required)
- `--tunnel-id <id>` — the `LC_OP_TUNNEL_ID` value (required)

**Startup logic (runs in foreground, prints to stderr):**

**Timing:** RemoteForward socket binding completes *before* `~/.ssh/rc` runs (confirmed from OpenSSH source: `ssh_init_forwarding()` executes before `ssh_session2_open()`). So by the time keepalive starts, the socket either exists (RemoteForward succeeded) or doesn't (it failed).

```
directory exists?
├── no → create directory with 0700 permissions
│        → "op-tunnel: ready — reconnect to activate"
│        → exit 0
└── yes → verify ownership (uid matches current user)
          ├── mismatch → "op-tunnel: directory owned by another user — aborting"
          │               → exit 1
          └── match → socket file exists?
                     ├── yes → net.DialTimeout("unix", path, 100ms)
                     │         ├── success: "op-tunnel: active via another session"
                     │         │            → exit 0 (other session's keepalive handles cleanup)
                     │         └── failure: stale socket
                     │                      → delete socket file
                     │                      → "op-tunnel: stale socket removed — reconnect to activate"
                     │                      → exit 0
                     └── no → RemoteForward succeeded (socket was just created)
                              → daemonize, monitor session
```

Cases that print "reconnect to activate" are expected only on first connection after install/reboot or after an unexpected disconnect. The exit code is 0 for informational messages, 1 for security errors (ownership mismatch).

**Daemonization (re-exec pattern):**

Go cannot `fork()` (goroutines + fork don't mix). Instead, the binary re-execs itself with a sentinel environment variable:

```go
if os.Getenv("_OP_TUNNEL_KEEPALIVE_DAEMON") == "" {
    // Foreground: validate, print status, then re-exec as daemon
    cmd := exec.Command(os.Args[0], os.Args[1:]...)
    cmd.Env = append(os.Environ(), "_OP_TUNNEL_KEEPALIVE_DAEMON=1")
    cmd.Stdin = nil
    cmd.Stdout = nil
    cmd.Stderr = nil
    cmd.SysProcAttr = &syscall.SysProcAttr{Setsid: true}
    if err := cmd.Start(); err != nil {
        fmt.Fprintf(os.Stderr, "op-tunnel-keepalive: %v\n", err)
        os.Exit(1)
    }
    // Parent exits; ssh.rc continues
    os.Exit(0)
}

// Daemon: monitor session PID
```

`Setsid: true` creates a new session so the daemon survives the SSH session's terminal teardown. `Stdin/Stdout/Stderr = nil` detaches from the terminal.

**Daemon monitoring loop:**

```go
for {
    time.Sleep(2 * time.Second)
    if err := syscall.Kill(sessionPID, 0); err != nil {
        // sshd session process is gone — clean up
        os.Remove(socketPath)
        os.Exit(0)
    }
}
```

`syscall.Kill(pid, 0)` is a POSIX signal-zero check — returns `nil` if the process exists, error if not. Works on both Linux and macOS. 2-second interval means stale sockets are cleaned up within 2 seconds of session end.

**Process visibility:**

```
$ ps aux | grep op-tunnel
alice  34567  op-tunnel-keepalive --session-pid 34500 --tunnel-id abc123def456
```

Easy to identify, troubleshoot, and kill if needed.

**Duplicate prevention:**

If a second SSH session connects with the same tunnel ID while the first is active:
1. RemoteForward fails (socket already exists, no `StreamLocalBindUnlink`).
2. Keepalive detects the socket is alive (another session), prints info message, exits without daemonizing.
3. The first session's keepalive continues monitoring.
4. Op commands in the second session use the existing socket (it forwards to the same local machine).

When the first session ends, its keepalive cleans up the socket. The second session's `op` commands will fail (expected — the tunnel was being shared, and the owner disconnected).

### 4. Updated `op-tunnel-setup`

The setup script runs at install time (Homebrew `post_install`). All sudo steps are removed.

**Steps:**

1. **Generate tunnel ID** — 32-char random hex, persisted to `~/.config/op-tunnel/tunnel-id`. Skipped if file exists. (Unchanged.)
2. **Generate ssh.config** — substitute `@@` placeholders, write to `~/.config/op-tunnel/ssh.config`. (Updated template paths.)
3. **Create op symlink** — `~/.local/bin/op` → `op-tunnel-client`. (Unchanged.)
4. **Create local socket directory** — Create with restrictive permissions and verify ownership: `old_umask=$(umask) && umask 077 && mkdir -p /tmp/op-tunnel-$USER/server && umask $old_umask`. Then verify the directory is owned by the current user (defense against pre-creation attacks). (New, replaces sudo mkdir.)
5. **Configure `~/.ssh/rc`** — Append the keepalive invocation line if not already present. (New, replaces sshd drop-in.)

**Removed steps:**
- `sudo mkdir -p /opt/op-tunnel/$USER` — replaced by `/tmp/` directory (user-writable).
- `sudo install op-tunnel-sshd.conf` — no longer needed.
- `sudo systemctl reload sshd` — no longer needed.

**`~/.ssh/rc` configuration (step 5):**

The installer appends the keepalive invocation to `~/.ssh/rc`, guarded by an idempotency check:

```bash
if ! grep -q 'op-tunnel-keepalive' "$HOME/.ssh/rc" 2>/dev/null; then
    mkdir -p "$HOME/.ssh"
    printf '\n# op-tunnel keepalive\n[ -n "$LC_OP_TUNNEL_ID" ] && op-tunnel-keepalive --session-pid "$PPID" --tunnel-id "$LC_OP_TUNNEL_ID"\n' >> "$HOME/.ssh/rc"
fi
```

The line is self-contained — no external script to source, no `$PPID` indirection through subshells.

**X11 forwarding caveat:** When `~/.ssh/rc` exists, sshd no longer runs `xauth` automatically for X11 forwarding. Op-tunnel targets terminal sessions; users needing X11 forwarding should add xauth handling to their `~/.ssh/rc` manually (see `sshd(8)` man page). This should be documented as a caveat in the README.

**Caveats printed after install:**
- SSH config Include instructions (unchanged).
- PATH warning if `~/.local/bin` not in PATH (unchanged).
- Note about `~/.ssh/rc` configuration (new): "Configured ~/.ssh/rc for automatic tunnel cleanup."

### 7. Multi-Session Behavior

**Different local machines → different `LC_OP_TUNNEL_ID`** → different socket files. Each gets its own keepalive daemon. No conflict.

**Same local machine, multiple sessions to same remote user:**
- First session: RemoteForward succeeds, keepalive daemonizes and monitors.
- Second session: RemoteForward fails (socket exists). Keepalive detects live socket, prints "active via another session", exits. Op commands work via the first session's socket.
- First session disconnects: keepalive cleans up socket. Second session's op commands stop working (tunnel owner disconnected).

This matches the previous `StreamLocalBindUnlink` "last wins" behavior, except it's "first wins" — the first session owns the socket until it disconnects. Both approaches are consistent with SSH agent forwarding semantics.

### 8. Platform Support

**Remote hosts (where `~/.ssh/rc` and keepalive run):**
- Linux: `$PPID` works, `syscall.Kill(pid, 0)` works, `/tmp/` is standard.
- macOS: Same — all POSIX. `/tmp/` is cleaned periodically (3-day default) but keepalive recreates dirs on each session.

**Local hosts (where op-tunnel-server runs):**
- macOS: `/tmp/` cleaned periodically. Server recreates its socket/dir on startup (brew service restart). `ServerAliveInterval 30` prevents cleanup of the socket while sessions are active.
- Linux: Same.

**Ubuntu/Debian `AcceptEnv` assumption:**
- Default `sshd_config` includes `AcceptEnv LANG LC_*` → `LC_OP_TUNNEL_ID` passes through.
- On systems without this default (RHEL, CentOS, custom sshd configs), `LC_OP_TUNNEL_ID` will be silently dropped. Op-tunnel won't activate on the remote — the client falls back to passthrough mode (local `op`).
- Doctor could detect this: if `LC_OP_TUNNEL_ID` is not set in the remote environment despite being configured, suggest adding `AcceptEnv LC_OP_TUNNEL_ID` to sshd_config.

## Backward Compatibility

This is a breaking change from the `further-streamline` branch. Users must re-run `op-tunnel-setup` on all machines. The Homebrew `post_install` hook handles this automatically.

Socket paths change from `/opt/op-tunnel/<user>/...` to `/tmp/op-tunnel-<user>/...`. Old directories under `/opt/op-tunnel/` can be manually removed. The old sshd drop-in at `/etc/ssh/sshd_config.d/op-tunnel.conf` is harmless if left in place but can be manually removed with `sudo rm /etc/ssh/sshd_config.d/op-tunnel.conf` (removing it requires sudo, which contradicts the zero-sudo goal — the spec does not attempt automatic cleanup).

## Security Model

The trust model remains equivalent to SSH agent forwarding:

- `/tmp/op-tunnel-<user>/` directories are `0700` (owner-only). The sticky bit on `/tmp/` prevents other users from removing or renaming the directory.
- Socket files are `0600` inside a `0700` directory — double-gated access.
- `LC_OP_TUNNEL_ID` is a 32-char random hex (128 bits) — not guessable, not enumerable from outside the `0700` directory.
- On multi-user systems, each user's sockets are isolated by their own directory.

**Directory ownership verification:** Since `/tmp/` is world-writable, an attacker could pre-create `/tmp/op-tunnel-alice/` before Alice's first connection. To prevent this, all code that creates or uses the directory must verify ownership after `mkdir -p`:

```go
info, err := os.Stat(dir)
if err != nil { return err }
stat := info.Sys().(*syscall.Stat_t)
if stat.Uid != uint32(os.Getuid()) {
    return fmt.Errorf("directory %s owned by uid %d, expected %d", dir, stat.Uid, os.Getuid())
}
```

This check is performed by:
- `op-tunnel-keepalive` on startup (before daemonizing) — see startup flowchart.
- `op-tunnel-server` when creating the server socket directory.
- `op-tunnel-setup` when creating the local socket directory (step 4).

Additionally, directory creation should minimize the window where permissions are open. Use `os.MkdirAll` with a restrictive umask, or `mkdir` followed immediately by `chmod 0700` in the same error-checked block.

**Compared to the `/opt/op-tunnel/` approach:** The `/tmp/` approach requires the ownership check (above) to match the security of `/opt/`. The `/opt/` approach had a root-owned base directory that prevented non-root users from creating entries, eliminating the squatting vector. The ownership verification provides equivalent protection in userspace.

**PID reuse risk:** The keepalive daemon monitors a PID. If the sshd session process exits and the same PID is reused by an unrelated process, the keepalive would not clean up the stale socket until the unrelated process also exits. This is extremely unlikely with modern PID allocation (32768+ PID space, sequential allocation) and the 2-second polling interval. Worst case: a stale socket persists until the PID-reusing process exits, at which point the keepalive cleans up.

## Known Limitations

- `op inject` is not yet supported.
- First SSH connection after install or reboot requires a reconnect (keepalive creates the directory, but RemoteForward already failed).
- On non-Ubuntu/Debian systems, `AcceptEnv LC_*` may not be in the default sshd_config. Users need to add it manually (or op-tunnel falls back to passthrough mode silently).
- Same-machine multi-session is "first wins" — second session uses the first session's tunnel. When the first disconnects, the second session's tunnel stops working.
- PID reuse (theoretical, see Security Model) could delay stale socket cleanup.
- X11 forwarding (`ssh -X`) breaks if `~/.ssh/rc` doesn't include xauth handling. Op-tunnel targets terminal sessions; users needing X11 should add the standard xauth block to their `~/.ssh/rc` (see `sshd(8)`). README should document this caveat.
