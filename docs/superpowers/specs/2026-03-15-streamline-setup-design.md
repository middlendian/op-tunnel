# Streamline Setup: Socket Security, Setup Automation, and Client Fixes

**Date:** 2026-03-15
**Status:** Draft
**Branch:** further-streamline

## Problem Statement

Three related issues with op-tunnel's current design:

1. **Socket security.** Sockets in `/tmp` are vulnerable to squatting attacks on multi-user systems. A malicious co-tenant can pre-create a listening socket at the predictable path, intercept op commands, and inject fake credentials. The sticky bit on `/tmp` prevents the legitimate user's sshd from unlinking the attacker's socket.

2. **Local `op` hangs.** When `op-tunnel-client` is symlinked as `~/.local/bin/op`, running `op` on the local machine (no SSH session) enters an infinite loop. `findRealOp()` compares resolved directories but misses symlink aliases in other directories, finding itself and `syscall.Exec`-ing in a loop.

3. **Setup friction.** Installation requires a separate manual `op-tunnel-setup` step, and the sshd configuration requires understanding why sudo is needed. The setup should run automatically at install time with clear explanations.

## Design

### 1. Directory Structure and Socket Paths

Replace `/tmp/` sockets with per-user directories under `/opt/op-tunnel/`:

```
/opt/op-tunnel/                              # root:root 0755
/opt/op-tunnel/<username>/                   # <username>:<group> 0700
/opt/op-tunnel/<username>/client/            # remote-forwarded sockets
/opt/op-tunnel/<username>/server/            # op-tunnel-server listening socket
```

Each user's directory is `0700`, preventing other users from listing, accessing, or creating files inside. This eliminates socket squatting entirely — no sticky-bit concerns, no path predictability issues.

**Server socket (local machine):**
`/opt/op-tunnel/<user>/server/op.sock`

Static name. op-tunnel-server listens here. Referenced by the right side of `RemoteForward`.

**Client socket (remote machine):**
`/opt/op-tunnel/<remote_user>/client/<LC_OP_TUNNEL_ID>.sock`

One socket per local-machine identity. Created by SSH's `RemoteForward`. The ID component ensures different local machines get different sockets on the same remote host.

**`LC_OP_TUNNEL_ID`:** A 32-character cryptographically random hex string (128 bits), generated at install time on the local machine. Persisted to `$XDG_CONFIG_HOME/op-tunnel/tunnel-id` (default `~/.config/op-tunnel/tunnel-id`). Preserved across updates; regenerated automatically if the file is missing (delete the file or the config directory to force regeneration). Baked into the generated `ssh.config`. Sent to the remote via `SetEnv`.

### 2. SSH Configuration

**`packaging/ssh.config.tmpl`** — a template with `@@`-delimited placeholders, substituted by `sed`:

```
# Auto-generated by op-tunnel-setup. Do not edit manually.
# Delete ~/.config/op-tunnel/ and re-run op-tunnel-setup to regenerate.
RemoteForward /opt/op-tunnel/%r/client/@@TUNNEL_ID@@.sock /opt/op-tunnel/@@LOCAL_USER@@/server/op.sock
SetEnv LC_OP_TUNNEL_ID=@@TUNNEL_ID@@
StreamLocalBindUnlink yes
ServerAliveInterval 30
```

At install time, `op-tunnel-setup` substitutes the placeholders using `sed`:

```sh
sed -e "s/@@TUNNEL_ID@@/$TUNNEL_ID/g" -e "s/@@LOCAL_USER@@/$USER/g" \
    ssh.config.tmpl > "$CONFIG_DIR/ssh.config"
```

The generated file is written to `$XDG_CONFIG_HOME/op-tunnel/ssh.config` (default `~/.config/op-tunnel/ssh.config`).

The user includes this in `~/.ssh/config` under specific host blocks:

```
Host my-server
    Include ~/.config/op-tunnel/ssh.config
```

**`packaging/op-tunnel-sshd.conf`** — simplified server-side drop-in:

```
AcceptEnv LC_OP_TUNNEL_ID
StreamLocalBindUnlink yes
```

`AcceptEnv` allows the env var through (many Debian/Ubuntu systems already accept `LC_*`, but the explicit drop-in ensures it works everywhere). `StreamLocalBindUnlink` allows sshd to clean up stale sockets on reconnect (no sticky-bit issue since the socket is in a user-owned `0700` directory).

### 3. Client Mode Logic

**Current:** Check `LC_OP_TUNNEL_SOCK` env var → tunnel if set, passthrough if not.

**New:**

```
LC_OP_TUNNEL_ID set?
├── yes → construct socket path from $USER + $LC_OP_TUNNEL_ID
│         → net.DialTimeout("unix", path, 100ms)
│         → success: tunnel mode (reuse open connection)
│         → failure: ERROR and exit (user expects tunnel)
└── no  → passthrough mode via findRealOp()
              → if real op not found: print "op not found" error and exit
```

When `LC_OP_TUNNEL_ID` is set, the user opted into tunneling. Connection failures are errors, not silent fallback. The 100ms timeout is a safety net; common failure cases (no socket file, stale socket) return instantly.

**Socket path construction:**

```go
filepath.Join("/opt/op-tunnel", os.Getenv("USER"), "client", os.Getenv("LC_OP_TUNNEL_ID")+".sock")
```

**Constraint:** The client constructs the path using `$USER`, which must match SSH's `%r` token (the remote login username). This holds in all standard configurations. Edge cases where `$USER` is overridden in the remote shell environment are not supported.

### 4. Fix `findRealOp` Infinite Loop

**Root cause:** `findRealOp()` skips directories matching the resolved binary's directory, but misses symlink aliases in other directories (e.g., `~/.local/bin/op` → `op-tunnel-client` in Homebrew Cellar).

**Fix:** Use `os.SameFile()` (device + inode comparison) instead of directory path comparison. This catches symlinks, hardlinks, and aliases from any directory. Inspired by ccache/distcc, which have used this pattern for 20+ years.

```go
func findRealOp() string {
    self, err := os.Executable()
    if err != nil {
        return ""
    }
    selfInfo, err := os.Stat(self)
    if err != nil {
        return ""
    }

    for _, dir := range filepath.SplitList(os.Getenv("PATH")) {
        if dir == "" {
            continue
        }
        candidate := filepath.Join(dir, "op")
        info, err := os.Stat(candidate)
        if err != nil || info.IsDir() || info.Mode()&0111 == 0 {
            continue
        }
        if os.SameFile(selfInfo, info) {
            continue
        }
        return candidate
    }
    return ""
}
```

### 5. Post-Install Setup (`op-tunnel-setup`)

Runs automatically as a Homebrew post-install hook. The same script handles both local and remote machines — op-tunnel is installed via Homebrew on both sides. All steps are idempotent and harmless on either side: generating a tunnel-id and ssh.config on a remote-only machine has no effect since no SSH client config references them.

**Non-privileged steps (no sudo):**

1. Generate `LC_OP_TUNNEL_ID` — 32-char random hex, persisted to `~/.config/op-tunnel/tunnel-id`. Skipped if the file already exists (preserves ID across updates). Delete the file to force regeneration on next run.
2. Generate `ssh.config` — substitute `@@` placeholders in `ssh.config.tmpl` via `sed`, write to `~/.config/op-tunnel/ssh.config`
3. Create `op` symlink — `~/.local/bin/op` → `op-tunnel-client`

**Privileged steps (sudo, with explanation):**

The script prints a brief explanation before prompting for sudo: "op-tunnel needs to create a private socket directory and configure sshd. This requires administrator access (one-time setup)."

If sudo fails or is denied, the script exits with a clear error: "Setup cannot continue without administrator access. The socket directory and sshd configuration are required for op-tunnel to function."

4. Create user directory and install sshd config:
   ```sh
   sudo mkdir -p /opt/op-tunnel/$USER
   sudo chown $USER /opt/op-tunnel/$USER
   sudo chmod 700 /opt/op-tunnel/$USER
   sudo install -m 644 op-tunnel-sshd.conf /etc/ssh/sshd_config.d/op-tunnel.conf
   ```

5. Reload sshd:
   ```sh
   # Linux (systemd)
   sudo systemctl reload sshd 2>/dev/null || sudo systemctl reload ssh
   # macOS
   sudo launchctl kickstart -k system/com.openssh.sshd
   ```

6. Create subdirectories (no sudo needed, user now owns parent):
   ```sh
   mkdir -p /opt/op-tunnel/$USER/{client,server}
   ```

**Colored caveats printed after install:**

- **SSH config** (yellow/bold): Instructions to add `Include ~/.config/op-tunnel/ssh.config` under a specific `Host` block in `~/.ssh/config`
- **PATH warning** (yellow/bold): If `~/.local/bin` is not in `$PATH`, instructions to add it

Colors use ANSI escape codes, suppressed when stdout is not a terminal.

**Idempotency:** All steps check existing state before acting. Re-running `op-tunnel-setup` preserves the existing tunnel ID and only regenerates `ssh.config` if the template or ID has changed. Directories are created only if missing. The sshd drop-in is overwritten to ensure it matches the current version.

### 6. `op-tunnel-doctor`

A diagnostic tool that checks op-tunnel health. Runs without arguments, prints a checklist with colored pass/fail indicators. Each failing check includes a recommended fix action.

**Checks and recommended actions:**

| # | Check | How | Fix |
|---|-------|-----|-----|
| 1 | Server running? | try connecting to server socket | `brew services restart op-tunnel` |
| 2 | Client socket active? | try connecting to client socket (only if `LC_OP_TUNNEL_ID` set) | Reconnect SSH session |
| 3 | Real `op` findable? | run `findRealOp()`, report path | `brew install --cask 1password-cli` |
| 4 | `~/.local/bin/op` symlink correct? | verify points to `op-tunnel-client` | `brew reinstall op-tunnel` |
| 5 | `~/.local/bin` in PATH? | check `$PATH` | Print shell-specific instructions to add it |
| 6 | SSH config includes op-tunnel? | scan `~/.ssh/config` for Include line | Print the `Host` + `Include` snippet to add |
| 7 | sshd drop-in installed? | check `/etc/ssh/sshd_config.d/op-tunnel.conf` exists | `brew reinstall op-tunnel` |
| 8 | Directory permissions correct? | verify `/opt/op-tunnel/$USER/` is `0700`, owned by user | `brew reinstall op-tunnel` |
| 9 | Config directory exists? | check `~/.config/op-tunnel/` with `tunnel-id` and `ssh.config` | `brew reinstall op-tunnel` |

`brew reinstall op-tunnel` re-runs `op-tunnel-setup` via the `post_install` hook, which recreates any missing state (symlink, directories, sshd config, ssh.config). Config files at `~/.config/op-tunnel/` and socket directories at `/opt/op-tunnel/$USER/` are outside the Homebrew prefix and survive uninstall/reinstall.

### 7. Shared Package: `oppath`

Extracts common logic used by client, server, and doctor:

- `BaseDir` constant: `/opt/op-tunnel`
- `ConfigDir() string` — returns `$XDG_CONFIG_HOME/op-tunnel` (default `~/.config/op-tunnel`)
- `ClientSocketPath(user, id string) string` — constructs client socket path
- `ServerSocketPath(user string) string` — constructs server socket path
- `FindRealOp() string` — the `os.SameFile`-based binary lookup (used by both client passthrough and server to find the real `op` binary, replacing `exec.LookPath("op")` which would find the client symlink)
- `EnvTunnelID` constant: `LC_OP_TUNNEL_ID`

### 8. Multi-Session Behavior

**Different local machines → different `LC_OP_TUNNEL_ID`** → different socket files in the same remote user's `client/` directory. These coexist without conflict.

**Same local machine, multiple sessions to same remote user** → same socket path. `StreamLocalBindUnlink` lets the latest session take over. Previous sessions' `op` commands fail with a connection error (the env var is set, so this surfaces as an explicit error rather than silent fallback).

This matches SSH agent forwarding behavior. For users who need multiple concurrent sessions to the same host, the recommended approach is SSH multiplexing (`ControlMaster`) or tmux on the remote.

## Backward Compatibility

This is a breaking change. Users upgrading must re-run `op-tunnel-setup` on both local and remote machines. The Homebrew post-install hook runs it automatically on the local machine. Remote machines need a one-time re-run.

## Security Model

The trust model remains equivalent to SSH agent forwarding, but with improved socket isolation:

- Per-user `0700` directories prevent cross-user socket access (squatting, interception, enumeration)
- Unpredictable socket filenames (random 32-char hex ID) add defense-in-depth
- Only `LC_OP_TUNNEL_ID` is forwarded via env var; allowlisted `OP_*` vars forwarded in the wire protocol
- Socket permissions are `0600` (owner-only), enforced by the `0700` parent directory

**Future hardening:** SSH signing key authentication for the socket connection (feasible with Go's `crypto/ssh` package and 1Password's SSH agent, estimated 2-4 days of work). Not in scope for this change.

## Known Limitations

- `op inject` is not yet supported
- Same-machine multi-session to the same remote user has "last wins" behavior
- NFS-mounted `/opt` would prevent socket creation (uncommon; `/opt` is typically local)
- Requires OpenSSH 7.3+ for `RemoteForward` with Unix sockets
